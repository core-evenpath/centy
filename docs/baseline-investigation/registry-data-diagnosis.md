## Top-cluster deep-dive

### Is the file auto-generated?

Yes. Header declares:
```
// AUTO-GENERATED by scripts/extract-block-registry-data.js
```

`git ls-files` confirms the file is committed (not gitignored). The generator lives at `scripts/extract-block-registry-data.js`. There is no `npm run`-style alias in `package.json`; the generator is run directly with `node scripts/extract-block-registry-data.js`.

### Error pattern

All 125 errors are identical-shape TS2353:

```
_registry-data.ts(275,28): error TS2353: Object literal may only specify known properties,
  and 'name' does not exist in type 'ServerSubVerticalData'.
```

They sit on lines 275-420 of the generated file, which is the `VERTICAL_SUBS_DATA: ServerSubVerticalData[]` literal.

### Root cause

Two declarations are out of sync within the generator's own output template:

1. **Interface declaration (line 23 of generated, line 129 of generator):**

   ```ts
   export interface ServerSubVerticalData {
     id: string;
     blocks: string[];
   }
   ```

2. **Emitted object literals (lines 275-420 of generated, extracted from vertical source files):**

   ```ts
   { id: 'new_vehicle_sales', name: 'New Vehicle Sales', industryId: 'automotive',
     blocks: [...] }
   ```

The generator uses `extractArray` to pull sub-vertical definitions verbatim from each vertical's `index.tsx` (e.g., `src/app/admin/relay/blocks/previews/automotive/index.tsx`). Those source definitions have included `name` and `industryId` for a long time — the consumer type `SubVerticalWithIndustry` in `src/lib/relay/module-analytics-derive.ts` relies on `industryId`, and callers in `analytics.ts:164` and `lookups.ts:40` use `as unknown as SubVerticalWithIndustry[]` casts specifically to paper over the interface-mismatch that the generator emits.

The interface declaration has never been updated to reflect the emitted shape. Every run of the generator reproduces the same 125 errors.

### Evidence the fix is surgical

- Generator is a single file (170 lines).
- The mismatch is 2 missing fields in an interface at template-embed time.
- Consumers already treat `industryId` as present (via casts); widening the interface doesn't force any caller change, only makes the casts unnecessary.
- `name` is not currently used by any caller (grep `\.name` on `SubVerticalWithIndustry`/`ALL_SUB_VERTICALS_DATA` returns nothing). Keeping it optional preserves safety.

## Diagnosis

**A — Generator staleness.** Interface declaration in the generator's output template is narrower than the object literals the generator extracts and emits. Mismatch has always been there since `_registry-data.ts` has existed.

Not B (type-definition drift — the interface lives inside the generator's own template, not an external file, so nothing else moved under it). Not C (TypeScript/tsconfig unchanged since last clean build). Not D.

## Fix complexity

**Surgical.** Two sub-edits:

1. Widen `ServerSubVerticalData` interface in the generator template to include `name?: string` and `industryId?: string` (both optional to preserve caller flexibility — even though every currently-emitted literal has them, marking optional avoids any brittleness in future subset shapes).
2. Regenerate `_registry-data.ts` by running the generator. No hand-edit to the output.

Optional follow-up inside this session after the fix lands and tsc drops: remove the `as unknown as SubVerticalWithIndustry[]` casts in `analytics.ts:164` and `lookups.ts:40`, which exist solely to work around the bug. Defer only if tsc doesn't match expectations.
