## Diagnosis: **B — type-definition mismatch (structural, pre-existing)**

### Evidence

**1. The file is auto-generated.** Header (lines 1–3):

```ts
// AUTO-GENERATED by scripts/extract-block-registry-data.js
// Server-safe data-only block registry (no React components).
// Regenerate with: node scripts/extract-block-registry-data.js
```

**2. Two incompatible type declarations exist for the same data.**

- Source type `SubVerticalDef` — `src/app/admin/relay/blocks/previews/_types.ts:22-28`:

  ```ts
  export interface SubVerticalDef {
    id: string;
    name: string;
    industryId: string;
    blocks: string[];
    genericBlocks?: string[];
  }
  ```

- Generated type `ServerSubVerticalData` — `_registry-data.ts:23-26` (emitted verbatim from the generator script):

  ```ts
  export interface ServerSubVerticalData {
    id: string;
    blocks: string[];
  }
  ```

**3. The generator extracts source arrays verbatim and dumps them into the stricter target type.** In `scripts/extract-block-registry-data.js`:

- Line 98: `const subsRaw = extractArray(src, `${v.prefix}_SUBVERTICALS`);` — pulls source text as-is
- Line 152–154: dumps raw into `VERTICAL_SUBS_DATA: ServerSubVerticalData[]`
- The generator *does* strip `preview` fields (lines 47–83, `stripPreviewField`) because `preview` is a React component incompatible with server-safe data. But it does **not** strip `name` or `industryId`.

**4. The mismatch is structural and pre-existing.** `git log -S "industryId" -- src/app/admin/relay/blocks/previews/_types.ts` → `7c36f26 Add block registry foundation files`. `industryId` has been on `SubVerticalDef` since the foundation commit. Every regeneration of `_registry-data.ts` since then would have produced the same mismatch.

**5. No drift in source files post-regeneration.** `git log ebf6e37..HEAD -- src/app/admin/relay/blocks/previews/` returns no commits. The last regeneration commit is `ebf6e37` (Phase 2 info.M02). Source vertical files are unchanged since; the committed generated file matches what the current source + current generator would produce.

### Why this rules out A (generator staleness)

"Generator staleness" means source shifted, file wasn't regenerated. Here, no source files shifted since the last regeneration. Re-running the generator as-is would produce the same broken output.

### Why this rules out C (environment drift)

- `git log -p 0ee6897 -- package.json tsconfig.json` shows no tsc/tsconfig/TS-version changes between Phase 2 close and HEAD (HEAD = `0ee6897`).
- Node 23.x and TS 5.9.3 are deterministic for TS2353 emission.
- The error is a valid structural type check, not an environmental quirk.

### Why this rules out D (something else)

The evidence chain — source type → generator → target type → error — is complete. No ambiguity.

## Why the claimed 401 baseline did not catch this

**Open question, not fully answered by this session.**

The 125-error cluster is structural and has existed since the foundation commit. Therefore every Phase 2 milestone that re-ran tsc over HEAD should have seen ≥125 errors on this file alone, yet Phase 2 close docs assert a stable 401. Three candidate explanations (no evidence to pick between them):

1. The 401 figure was computed against a different file state — e.g., an older (pre-name/pre-industryId) version of `_registry-data.ts`, possibly locally regenerated from different sources. If so, the 401 was never a real measurement of committed `main`.
2. The 401 was estimated or copy-forwarded through multiple retrospectives without actual re-measurement (despite the stated discipline in `docs/engine-rollout-phase2/tuning.md` §Gate-session meta note).
3. A measurement environment existed that did not trigger TS2353 on these objects — e.g., different tsconfig `strict` settings at some past point. Current tsconfig does trigger it; no change in `tsconfig.json` between `0ee6897` and HEAD, so this would require the drift to predate Phase 2 close.

The outcome doc (`outcome.md`) notes that even after fixing the 125-error cluster, the remaining error count may land *below* 401 (455 − 125 = 330, minus any industryId-derived additions). That itself is a finding: the historical 401 was not a true floor.

## Estimated fix complexity: **surgical** (<60 min expected)

The generator is 170 lines and the fix has two parts:

1. **Hardcoded path** — line 6: `const PREVIEWS_DIR = '/home/user/centy/src/app/admin/relay/blocks/previews';`. This path does not exist on the current machine (macOS dev env). Fix: compute relative to `__dirname`. Required for the generator to run at all in this repo.

2. **Widen `ServerSubVerticalData`** — line 129–132 of the generator's output template. Expand to:

   ```ts
   export interface ServerSubVerticalData {
     id: string;
     name: string;
     industryId: string;
     blocks: string[];
     genericBlocks?: string[];
   }
   ```

   This makes the emitted type a superset that accepts what source files actually contain. The fields are meaningful data (human-readable name, vertical linkage) that server consumers may want anyway. Alternative — stripping `name`/`industryId` in the generator (like it strips `preview`) — loses that data; rejected.

3. **Regenerate** — run `node scripts/extract-block-registry-data.js`. Inspect diff. Expected: the type declaration changes; `VERTICAL_BLOCKS_DATA` rows are unchanged; `VERTICAL_SUBS_DATA` rows are unchanged (they already have `name` and `industryId`).

4. **Re-measure** — expect tsc to drop to ~330 (all 125 TS2353 errors resolved, no new errors introduced by widening the type since widening only relaxes constraints).
