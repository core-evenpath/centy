# MR-1.M01 — Retrieval Strategy Decision

**Date:** 2026-04-21
**Status:** locked
**Decides:** the technical architecture for MR-1.M02–M05 and MR-2
**Source evidence:** `docs/relay-mission/PHASE_0_RECON.md` + direct reading of `src/lib/relay/orchestrator/signals/rag.ts`, `src/ai/fireRagSetup.ts`, `src/actions/modules-actions.ts`, `src/actions/business-persona-actions.ts`, `src/lib/business-persona-types.ts`, `src/lib/modules/types.ts`

---

## Summary

MR-1 builds the writer that puts module items, business persona, and partner-uploaded docs into a vector store that the orchestrator can retrieve from. Phase 0 confirmed no such writer exists for relay partners; the `centy_documents` collection is thesis-only. This document locks nine decisions so MR-1.M02–M05 can execute without design debates. The key calls: separate `relayRetrieval` collection tree per partner and kind; hybrid retrieval for items (structured filter + vector); pure vector for persona and docs; one doc per item; three persona chunk kinds (faq / policy / identity); reuse `indexToFirestore`, `gemini-embedding-001`, and the PDF chunker unchanged; extend `loadRagSignal` rather than replace it.

---

## Decisions

### D1 — Storage architecture

**Context:** The existing `centy_documents` collection is used exclusively by the thesis-docs feature. Phase 0 confirmed the relay orchestrator queries `centy_documents` with `where: { partnerId }` and gets empty results. We need a new home for relay-indexed data.

**Options considered:**
- **Global `centy_documents` with `kind` + `partnerId` filter** — reuses existing Firestore index; risk: pollutes thesis retrieval, single collection limit at scale, harder to inspect per-partner
- **New `relayRetrieval/{partnerId}/{kind}/*` subcollection** — isolated; per-kind queries are Firestore collection scans; MR-5 cleanup is `deleteCollection(relayRetrieval/{pid}/items)` which is clean; admin inspector (MR-5.M02) can query per-partner with a known path
- **Enhance `moduleItems` in-place** — add `embedding` field directly on each item doc — avoids a new collection, but can't query by semantic similarity without collection-level vector index; mixes write-path concerns
- **New flat `relayIndex` collection with `partnerId` + `kind` fields** — single vector index, simpler; one collection group index required; harder to scope deletes per partner

**Decision:** `relayRetrieval/{partnerId}/items/{itemId}`, `relayRetrieval/{partnerId}/persona/{sectionId}`, `relayRetrieval/{partnerId}/docs/{chunkId}` — separate subcollection per kind per partner.

**Reasoning:** Isolation from `centy_documents` prevents thesis-doc retrieval contamination. Per-kind subcollections make MR-5 deletion clean (deactivate item → delete `relayRetrieval/{pid}/items/{itemId}`, no scatter query needed). The admin retrieval inspector in MR-5.M02 gets a clean `relayRetrieval/{pid}` path to count across all kinds. The Firestore vector index is created once on `relayRetrieval` with a collection-group scope.

### D2 — Item retrieval strategy

**Context:** `ModuleItem` has structured fields (`name`, `category`, `price`, `isActive`, `tags` via `fields`) and an existing `ragText` plain-text blob (generated by `generateRAGText` in `modules-actions.ts`, stored inline on the item doc but not vectorized). Pure vector retrieval for catalog queries underperforms: a user asking "Ethiopian coffee under ₹500" needs both semantic matching (Ethiopian → Yirgacheffe) and structured filtering (price, isActive).

**Options considered:**
- **Pure vector** — embed the ragText, retrieve by cosine; misses exact-match price/category constraints; inexpensive to implement
- **Pure structured** — Firestore `where` filters on `category`, `price`, `isActive`; no semantic flexibility; "gift for coffee lover" returns nothing
- **Hybrid: structured pre-filter + embedding rerank** — filter on `isActive=true` at query time; retrieve top-N candidates; rerank by embedding similarity on the filtered set; best precision

**Decision:** Hybrid for items — filter on `isActive: true`, retrieve k=8 candidates from the vector index, return top-4 by score.

**Reasoning:** Inactive items must never surface (compliance requirement; a deactivated item appearing in responses is worse than a miss). Embedding-only retrieval can't guarantee this without post-filtering that wastes token budget. The vector index on `relayRetrieval/{pid}/items` already includes `isActive` as a structured field (see D8), enabling Firestore's `where` + vector `findNearest` in one query. k=8 → top-4 gives enough candidates for reranking without inflating prompt size.

### D3 — Item chunking granularity

**Context:** `ModuleItem` records are short (name ~30 chars, description ~100–500 chars, a few tag-like `fields`). The `ragText` field in the item document concatenates name + description + category + key fields into a single text blob. `indexToFirestore` in `fireRagSetup.ts` (line 75) iterates over a `string[]` and writes one Firestore doc per string.

**Options considered:**
- **One doc per item** — use `ragText` as the chunk text; simple; natural deletion boundary (MR-5: delete doc when item deleted)
- **Multi-chunk per item** — split long descriptions; rarely needed at catalog scale (items are short); complicates deletion (need to track chunk count per item)
- **Group items by category** — one doc per category blob; loses per-item granularity; can't delete individual item chunks

**Decision:** One retrieval doc per item, using the existing `ragText` field as the chunk text.

**Reasoning:** Catalog items are short — splitting them creates noise, not signal. `ragText` already encodes the right searchable text (Phase 0: generated by `generateRAGText` in `modules-actions.ts:834`). One-doc-per-item makes MR-5's delete path trivially `deleteDoc(relayRetrieval/{pid}/items/{itemId})` with no chunk tracking. The `itemId` as the doc ID enables idempotent upserts (re-index = overwrite same doc ID).

### D4 — Persona chunking granularity

**Context:** `BusinessPersona` (from `src/lib/business-persona-types.ts`) has four major sections: `identity` (name, hours, contact, address), `personality` (description, tone, USPs), `profile` (target audience, common queries), and `knowledge` (FAQs, policies, products). FAQs are a `FrequentlyAskedQuestion[]` flat list. Policies are a structured `BusinessPolicies` object with shipping, returns, cancellation, etc. as separate fields.

**Options considered:**
- **One blob per full persona** — fast to index; terrible at retrieval (every FAQ query returns the whole blob)
- **One doc per FAQ entry** — natural retrieval unit; a customer asking a question maps to one FAQ answer
- **One doc per policy type** — shipping policy is a distinct retrievable unit from returns policy
- **Mixed: one per FAQ, one per policy, one identity blob** — covers all factual retrieval patterns

**Decision:** Three chunk kinds for persona:
- `kind: 'faq'` — one chunk per `knowledge.faqs[]` entry; text = `Q: {question}\nA: {answer}`
- `kind: 'policy'` — one chunk per non-empty policy in `knowledge.policies`; text = `{policy_type}: {policy_text}`
- `kind: 'identity'` — one chunk per partner; text = `{name} | {description} | {USPs} | hours: {hours} | contact: {email} | {phone}`

**Reasoning:** FAQs are the primary factual retrieval unit — a customer question maps to one FAQ answer, not a blob. Separate policy chunks prevent shipping-policy text from bleeding into FAQ retrieval. The identity chunk handles "what do you sell" / "when are you open" holistically and is small enough (< 512 tokens) to not need further splitting. `sectionId` for FAQs = `faq_{index}`, for policies = `policy_{type}`, for identity = `identity` — stable IDs for idempotent re-indexing.

### D5 — Doc chunking

**Context:** `indexPdfFile` in `src/ai/fireRagSetup.ts` (line 45) uses `RecursiveCharacterTextSplitter` with `chunkSize: 1024 * 2` (2KB), `chunkOverlap: 128`. This is the only existing chunker. Relay partners upload PDFs via the vault system (`vaultFiles`). No relay-side doc indexer exists yet (confirmed Phase 0).

**Options considered:**
- **Reuse `indexPdfFile` verbatim** — call it with `relayRetrieval/{pid}/docs` as the collection; zero new code for chunking logic; proven in thesis-docs
- **Fork and tune** — adjust chunk size for relay (product catalogs may have denser tables); adds maintenance burden

**Decision:** Reuse `indexPdfFile` verbatim, passing `relayRetrieval/{pid}/docs` as the collection name instead of `RAGINDEX_COLLECTION_NAME`.

**Reasoning:** The 2KB/128 overlap split is standard for dense prose and works well for policy documents and product catalogs. Forking before we have real-partner eval data (MR-4) is premature optimisation. The only change MR-1.M04 makes is the collection name argument. If MR-4 surfaces chunking failures, tune then.

### D6 — Embedding model

**Context:** `fireRagSetup.ts:14` defines `embedder = googleAI.embedder("gemini-embedding-001", { outputDimensionality: 768 })`. `firestoreRetriever` uses this embedder with `distanceMeasure: "COSINE"`. The same model is used for thesis-docs retrieval today.

**Options considered:**
- **Reuse `gemini-embedding-001` (768-dim, COSINE)** — single model across all collections; consistent vector space; no new setup
- **Use a different model** — no clear benefit; would require a separate Firestore vector index with different dimensionality; can't compare scores across collections

**Decision:** Reuse `gemini-embedding-001` with 768-dim COSINE.

**Reasoning:** Changing the embedding model requires a new Firestore vector index. There's no evidence that a different model would improve relay retrieval quality at this stage. Consistent vector space across `centy_documents` and `relayRetrieval` means future cross-collection queries remain possible. Revisit at MR-4 eval if recall is poor.

### D7 — Retrieval interface

**Context:** `loadRagSignal(ctx, intent)` in `src/lib/relay/orchestrator/signals/rag.ts` (line 64) already retrieves from `centy_documents` with a partner-scoped filter. Phase 0 confirmed this function is complete and working — it returns empty results only because hop 04 has no data. The current `RagSignal` type has `{ used, query, chunks[], reason }`. The README's MR-2.M01 now calls for extending this rather than building a new abstraction.

**Options considered:**
- **Extend `loadRagSignal` with `kind` parameter** — non-breaking (add optional `kind?: 'items' | 'persona' | 'docs'`); called three times per turn by the orchestrator; per-kind results compose into `ragSourceBreakdown`; minimal surface change
- **New `retrievePartnerContext(partnerId, query, opts)` wrapper** — the original MR-2.M01 proposal; builds an abstraction layer over `loadRagSignal`; adds indirection without clear benefit at this stage
- **Multiple parallel `loadRagSignal` calls unchanged** — call current function three times against three different collections; no signature change; loses per-kind tagging on results

**Decision:** Extend `loadRagSignal` to accept an optional `collectionPath?: string` parameter (defaulting to `RAGINDEX_COLLECTION_NAME`). MR-2 calls it three times per relevant turn — once per kind — with `relayRetrieval/{pid}/items`, `relayRetrieval/{pid}/persona`, `relayRetrieval/{pid}/docs`. Results merge into `ragSourceBreakdown: { items, persona, docs }`.

**Reasoning:** The current function body handles all the conditional logic (skip transactional intents, handle empty results, error-safe). Adding a `collectionPath` parameter changes one line (`firestoreRetriever(RAGINDEX_COLLECTION_NAME)` → `firestoreRetriever(collectionPath ?? RAGINDEX_COLLECTION_NAME)`). This is the smallest possible surface change that unblocks MR-2. Building a new abstraction before MR-1 data exists is premature — MR-2 can decide if a wrapper is warranted after validating end-to-end retrieval.

### D8 — Per-kind retrieval doc schemas

**Context:** `indexToFirestore` currently writes `{ fileId, partnerId, embedding: FieldValue.vector(embedding), text }`. The `firestoreRetriever` reads `contentField: "text"` and `vectorField: "embedding"`. Any new schema must include at least `text` and `embedding` to be compatible with the existing retriever.

**Decision:**

```
relayRetrieval/{partnerId}/items/{itemId}
{
  text: string,              // ragText from ModuleItem (generated by generateRAGText)
  embedding: vector(768),    // gemini-embedding-001
  partnerId: string,
  itemId: string,
  moduleId: string,
  kind: 'item',
  name: string,              // denormalised for MR-5 inspector display
  category: string,          // used for structured pre-filter in D2
  price: number | null,      // for future range-filter use
  currency: string,
  isActive: boolean,         // REQUIRED — used for pre-filter in D2
  indexedAt: string,         // ISO timestamp — MR-5 inspector shows freshness
}

relayRetrieval/{partnerId}/persona/{sectionId}
{
  text: string,              // formatted as "Q: ... \nA: ..." or "policy_type: ..." or identity blob
  embedding: vector(768),
  partnerId: string,
  sectionId: string,         // 'faq_0', 'faq_1', 'policy_shipping', 'identity'
  kind: 'faq' | 'policy' | 'identity',
  indexedAt: string,
}

relayRetrieval/{partnerId}/docs/{chunkId}
{
  text: string,              // PDF chunk from RecursiveCharacterTextSplitter
  embedding: vector(768),
  partnerId: string,
  fileId: string,            // vaultFiles doc ID — for deletion when file removed
  chunkIdx: number,          // 0-based chunk index within the file
  kind: 'doc',
  indexedAt: string,
}
```

**Reasoning:** `isActive` on items is required to support the hybrid pre-filter (D2). `indexedAt` on every doc enables MR-5's freshness warnings without a separate tracking collection. `kind` on every doc enables future cross-kind queries and MR-5's count breakdown. `sectionId` stable IDs enable idempotent re-indexing (persona re-save = upsert same sectionId). `fileId` on docs enables MR-5's delete-on-vault-remove path.

---

### D9 — Explicit out-of-scope for MR-1

What this strategy does **not** cover:

- **Delete / update / staleness tracking** — when an item is deactivated, the `relayRetrieval/{pid}/items/{itemId}` doc stays until MR-5.M01 wires the delete path. MR-4 eval will surface any stale-item problems.
- **Cross-kind retrieval orchestration** — the orchestrator calls `loadRagSignal` three times per turn; how to weight items vs persona vs docs results is MR-2's decision.
- **Admin retrieval inspector** — showing per-partner indexed counts + freshness warnings → MR-5.M02.
- **Real-partner acceptance eval** — 20-question eval with real SKUs and real FAQs → MR-4.
- **Flow-editor plumbing** — admin system templates connecting to orchestrator → MR-3.
- **Widget bundle, embed path** — P5, frozen until MR-4.
- **`ragText` inline field on moduleItems** — this field stays as-is. MR-1.M02 reads it as the chunk text for indexing (`item.ragText`), but does not remove it from the item document. It remains a useful local cache; removing it is MR-5 territory.
- **`relayKnowledge` config / exclusion lists** — `relay-knowledge-actions.ts` manages `excludedVaultDocIds` for the partner; the doc indexer should check this config before indexing a vault file. This is a guard condition in MR-1.M04, not a strategy question.

## Unlocked by this doc

Handoff to MR-1.M02 — item indexer:
- Collection path: `relayRetrieval/{partnerId}/items/{itemId}`
- Schema: see D8 items schema
- Text source: `item.ragText` (already computed by `generateRAGText` in `modules-actions.ts:834`)
- Embedding: `gemini-embedding-001` via `indexToFirestore` (or direct `ai.embed` call, same path)
- Hook points: after `adminDb.collection(...).doc(itemId).set(item)` in `createModuleItemAction` (line ~860) and after the update write in `updateModuleItemAction`
- Retrieval pre-filter: `isActive: true` stored on the doc; Firestore `where` before vector query

Handoff to MR-1.M03 — persona indexer:
- Collection path: `relayRetrieval/{partnerId}/persona/{sectionId}`
- Schema: see D8 persona schema
- Chunk kinds: `faq_{n}` per FAQ entry, `policy_{type}` per non-empty policy, `identity` once
- Text format: FAQs as `Q: {q}\nA: {a}`, policies as `{type}: {text}`, identity as name + description + USPs + hours + contact
- Hook point: after the Firestore write in `saveBusinessPersonaAction` (line ~266)

Handoff to MR-1.M04 — doc indexer:
- Reuse `indexPdfFile` from `fireRagSetup.ts:45` with `relayRetrieval/${partnerId}/docs` as collection
- Schema: see D8 docs schema (add `kind: 'doc'`, `chunkIdx`, `indexedAt` to what `indexToFirestore` writes)
- Guard: check `excludedVaultDocIds` from `relay-knowledge-actions.ts` before indexing
- New route: `src/app/api/relay/upload-doc/route.ts`

Handoff to MR-2.M01 — extend `loadRagSignal`:
- Extend signature: add optional `collectionPath?: string` parameter (see D7)
- Call three times per relevant turn with per-kind collection paths
- Return `ragSourceBreakdown: { items: number, persona: number, docs: number }` by counting chunks per call
- The `isActive` filter for items is a Firestore `where` option passed to `ai.retrieve`, not post-processing

## Open questions NOT answered here

1. **`relayKnowledge` exclusion guard for docs** — does `updateRelayDocExclusionsAction` in `relay-knowledge-actions.ts` write synchronously? If a partner excludes a doc and the indexer hasn't seen the update yet, it might index an excluded doc. The guard in MR-1.M04 should read `excludedVaultDocIds` at index time, not trust a cached value. Flag for MR-1.M04 implementer.

2. **Persona re-index invalidation** — on `saveBusinessPersonaAction`, all existing persona chunks should be deleted before re-indexing (otherwise stale FAQ chunks accumulate). The simplest implementation is `deleteCollection(relayRetrieval/{pid}/persona)` then re-index. MR-5 owns the durability story but MR-1.M03 needs to decide: delete-then-rewrite (safe, slightly slow) or upsert-by-sectionId (faster, risks stale entries if FAQ order changes). Recommend delete-then-rewrite for correctness; flag for implementer.

3. **`price` field on items** — `ModuleItem.price` is `number | undefined`. The schema stores it as `number | null`. Items with no price still get indexed (price not required for retrieval). If a future MR wants range queries ("products under ₹500"), a `priceNumeric` field should be added as a Firestore indexable field. Not needed for MR-1; surfacing here for MR-4 eval design.
